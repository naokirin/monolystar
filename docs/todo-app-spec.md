# 「やることだけ」ToDo管理アプリ 要件・仕様書

## 1. 概要

### 1.1 コンセプト
既存のToDoリストでは「自分の管理したい方法」で管理しきれないという課題を解決するため、以下を軸にしたシンプルなタスク管理アプリ。

- 思いついた瞬間にタスクをさっと追加できる
- 優先度と日時範囲をもとに自動で並び替えられ、上から順にこなすだけで優先タスクから処理できる
- ワンショットタスクと、習慣化すべき定期タスクの両方を一元管理できる

### 1.2 動作環境
- 単一HTMLファイル（ブラウザで完結するWebアプリ）
- データはブラウザ側のストレージに永続化（タスク本体・定期タスクの完了履歴・通知設定をそれぞれ別キーで保存）
- 通知機能はブラウザの `Notification API` を利用し、**アプリのタブを開いている間のみ**動作する（バックグラウンド/タブを閉じた状態では通知されない）
- **複数デバイス間のデータ共有**は、開発者側のサーバやクラウドサービス連携（OAuth・アプリ審査）を前提としない方式で実現する（詳細は第8章）

### 1.3 複数デバイス共有の設計方針

以下の制約を満たす同期方式のみを採用候補とする。

| 方針 | 内容 |
|---|---|
| データ主権はユーザー側 | 開発者がユーザーデータをホスト・保管・削除対応する責任を負わない |
| サーバレス優先 | 自前バックエンド（ユーザーアカウント、DB、認証基盤）を設けない |
| 審査・登録を避ける | Google Drive API / Dropbox API 等の OAuth アプリ登録、利用審査、ストア審査に依存しない |
| フロントエンド完結を維持 | 同期ロジックはブラウザ内で完結し、配布物は引き続き単一 HTML を基本とする |

この方針により、公開後に「利用者データの管理義務」「プライバシーポリシー上の第三者提供」「API 利用審査の維持」といった運用負荷を最小化する。

## 3. データモデル

### 3.1 タスク（Task）

ワンショットタスクと定期タスクのテンプレートは同一の構造で管理する（`recurrence.type` の値で区別）。

| フィールド | 型 | 説明 |
|---|---|---|
| `id` | string | タスクの一意なID |
| `title` | string | タイトル（必須・最大200文字） |
| `detail` | string | 詳細メモ（任意） |
| `priority` | `"must"` \| `"should"` | 優先度。`must`=必須、`should`=できれば |
| `category` | string | カテゴリ（任意・自由入力） |
| `startDate` | string (`YYYY-MM-DD`) \| null | 開始日。ワンショットでは「この日から表示」、定期タスクでは「有効期間の開始日」 |
| `startTime` | string (`HH:MM`) \| null | 開始時刻（任意） |
| `endDate` | string (`YYYY-MM-DD`) \| null | 終了日。ワンショットでは「この日まで（締切）」、定期タスクでは「有効期間の終了日」 |
| `endTime` | string (`HH:MM`) \| null | 終了時刻（任意） |
| `recurrence` | object | 繰り返し設定（3.2参照） |
| `completed` | boolean | 完了フラグ（**ワンショットタスクのみ使用**。定期タスクは別管理） |
| `completedAt` | number \| null | 完了日時（UNIXタイムスタンプ、ms） |
| `createdAt` | number | 作成日時（UNIXタイムスタンプ、ms） |
| `updatedAt` | number | 最終更新日時（UNIXタイムスタンプ、ms）。同期時の競合解決に使用 |
| `deletedAt` | number \| null | 論理削除日時。同期時に他端末へ削除を伝播するために使用 |
| `notifiedStart` | boolean | 開始通知を送信済みかどうか |
| `notifiedEndSoon` | boolean | 締切接近通知を送信済みかどうか |

### 3.2 繰り返し設定（recurrence）

| type | 追加フィールド | 意味 |
|---|---|---|
| `none` | なし | 繰り返さない（ワンショットタスク） |
| `daily` | なし | 毎日 |
| `weekly` | `weekday`（0=日〜6=土） | 毎週指定曜日 |
| `biweekly` | `weekday`（0=日〜6=土） | 隔週指定曜日（`startDate` を基準週として2週間おきに判定） |

### 3.3 定期タスクの完了履歴（Completions）

定期タスクは「テンプレート」のため、`completed` フラグでは管理せず、**日付ごとの完了記録**を別データとして保持する。

- キー形式: `{taskId}__{YYYY-MM-DD}`
- 値: `{ "at": number }`（完了記録日時、ms）。同期マージ時は `at` が新しい方を採用

これにより、同じ定期タスクが「今日は完了、明日はまた未完了」という状態を表現できる。

### 3.4 通知設定（Prefs）

| フィールド | 型 | 説明 |
|---|---|---|
| `notif` | boolean | ブラウザ通知を有効化しているかどうか |
| `updatedAt` | number | 最終更新日時（ms）。同期時の競合解決に使用 |

### 3.5 永続化キー

| キー | 内容 |
|---|---|
| `tasks-v1` | タスク配列（JSON） |
| `completions-v1` | 定期タスクの完了履歴（JSON） |
| `prefs-v1` | 通知設定など（JSON） |
| `sync-meta-v1` | 同期用メタデータ（3.6参照。未設定時は同期機能を使わない） |

ローカルストレージ上のデータはすべて個人データとして扱う。開発者側のサーバには送信しない。

### 3.6 同期用メタデータ（SyncMeta）

複数デバイス同期を有効にした場合に追加で保持する。

| フィールド | 型 | 説明 |
|---|---|---|
| `deviceId` | string | この端末を識別する UUID（初回起動時に生成） |
| `lastSyncedAt` | number \| null | 最後に同期に成功した日時（ms） |
| `syncMode` | `"file"` \| `"webdav"` \| null | 有効な同期方式。`null` は同期オフ |
| `fileHandle` | （非永続） | File System Access API で取得したファイルハンドル。セッション内のみ保持し localStorage には保存しない |
| `webdav` | object \| null | WebDAV 接続設定（8.3参照。パスワードは端末の sessionStorage のみに保持） |

### 3.7 同期ファイル形式（エクスポート／ファイル同期／WebDAV 共通）

手動エクスポート・ファイル同期・WebDAV 同期で共通の JSON スキーマを用いる。

```json
{
  "schemaVersion": 1,
  "exportedAt": 1710000000000,
  "deviceId": "uuid-of-exporting-device",
  "tasks": [ /* Task 配列 */ ],
  "completions": { /* Completions オブジェクト */ },
  "prefs": { /* Prefs オブジェクト */ }
}
```

各 `Task` には同期のため `updatedAt`・`deletedAt` を含める（3.1参照）。

`completions` の各キーは値として `{ "at": number }` を持つ（3.3参照）。

## 4. 画面構成・機能仕様

### 4.1 ヘッダー
- アプリ名「やることだけ」とタグライン
- 通知ON/OFFボタン（🔔）

### 4.2 クイック追加バー
- テキスト入力＋「追加」ボタン（Enterキーでも追加可能）
- 入力のみで以下のデフォルト値でタスクを作成
  - 優先度: `should`（できれば）
  - 日時範囲: なし
  - カテゴリ: なし
  - 繰り返し: `none`
- 「＋ 詳細・日時・優先度を設定して追加」リンクから詳細フォームを開ける（入力済みのタイトルは引き継がれる）

### 4.3 詳細入力フォーム（追加・編集共通のモーダル）
以下の項目を設定できる。新規作成・既存タスクの編集の両方で同じフォームを使用する。

1. タイトル（必須）
2. 詳細（任意・複数行）
3. 優先度（必須／できれば の2択）
4. カテゴリ（自由入力。既存カテゴリを入力候補として表示）
5. 繰り返し（なし／毎日／毎週／隔週）
   - 毎週・隔週を選択すると曜日選択UIが出現
6. 開始日・開始時刻（任意）
7. 終了日・終了時刻（任意）
8. 編集時のみ「削除」ボタンを表示（確認ダイアログ後に削除）

ラベルは繰り返し設定の有無によって以下のように変化する。
- ワンショット時: 「開始日（この日から表示）」「終了日（この日まで／締切）」
- 定期タスク時: 「期間の開始日」「期間の終了日」（＝その定期タスクが有効な期間）

### 4.4 タブ構成

| タブ | 表示内容 |
|---|---|
| 今日 | 「今日やるべきタスク」一覧（4.5参照） |
| すべて | 未完了のワンショットタスクを開始日昇順で一覧 |
| 完了 | 完了済みワンショットタスクを完了日時の新しい順で一覧。タップで未完了に戻せる |
| 定期タスク | 登録済みの定期タスクテンプレート一覧。編集・削除が可能。今日分が完了済みの場合はラベル表示 |

各タブには対象件数をカウント表示する。

### 4.5 「今日」タブの抽出・並び替えロジック

**抽出条件**

- ワンショットタスク: 以下をすべて満たすもの
  - 未完了（`completed = false`）
  - `startDate` が未設定、または今日以前
  - `endDate` が未設定、または今日以降
- 定期タスク: 以下をすべて満たすもの
  - 今日がその繰り返しルールに該当する日（毎日／指定曜日／隔週の該当週かつ指定曜日）
  - 今日の日付の完了記録（Completions）が存在しない
  - 有効期間（`startDate`〜`endDate`）内である場合のみ対象

**並び替え条件（優先順）**

1. 優先度（`must` が `should` より上位）
2. 締切の近さ（`endDate`＋`endTime` が早いものが上位。締切未設定は最下位扱い）
3. 開始日の早さ（締切が同条件の場合のタイブレーク）

これにより「上から順にこなせば優先度の高いもの・締切が近いものから処理できる」状態を実現する。

### 4.6 完了操作
- 丸いチェックボタンをタップすると即座に完了／未完了を切り替える
- ワンショットタスク: `completed` フラグと `completedAt` を更新
- 定期タスク: 当日分の完了記録（Completions）をトグル
- 完了時は取り消し線とフェード表示で視覚的にフィードバック

### 4.7 タスクカードの表示要素
- 優先度を示す左端のカラーストライプ（必須＝赤／できれば＝黄）
- タイトル
- バッジ: 優先度／カテゴリ（設定時）／日時範囲（設定時）／繰り返しルール（今日タブ・定期タスクタブのみ）
- 締切24時間以内のタスクは日時バッジを警告色で強調表示
- 詳細メモがあれば本文下に表示
- カードタップで編集フォームを開く（チェック部分のタップは完了切替のみで編集は開かない）

## 5. 通知仕様

### 5.1 有効化
- ヘッダーの🔔ボタンから `Notification.requestPermission()` を呼び出してユーザーに許可を求める
- 許可された場合のみ通知設定を保存し、以降は60秒間隔で判定処理を実行

### 5.2 開始通知
- 対象: `startDate` が設定されたワンショットの未完了タスク
- 条件: 現在時刻が「開始日＋開始時刻（未設定時は当日00:00）」を過ぎており、かつ未通知（`notifiedStart = false`）
- 通知後は `notifiedStart = true` を記録し、再通知しない

### 5.3 締切接近通知
- 対象: `endDate` が設定されたワンショットの未完了タスク
- 条件: 「終了日＋終了時刻（未設定時は当日23:59）」までの残り時間が24時間以内、かつ未通知（`notifiedEndSoon = false`）
- 通知後は `notifiedEndSoon = true` を記録し、再通知しない

### 5.4 制約事項
- 通知はアプリのタブを開いている間のみ判定・発火する（Service Workerによるバックグラウンド通知には対応していない）
- 定期タスクは現状、開始・締切通知の対象外（テンプレートのため都度の日時を持たないため）

## 6. 非機能要件・UI方針

- レスポンシブ対応（モバイル幅を基準に最大640px幅でセンタリング）
- フォントは見出しに明朝体（Shippori Mincho）、本文にゴシック体（Zen Kaku Gothic New）、日時表記にモノスペース（Roboto Mono）を使用し、紙の伝票・チェックリストのような質感を意図したデザイン
- 操作フィードバックとしてトースト通知（画面下部に短時間表示）を使用
- 空の状態（タスクなし）では、各タブごとに案内文を表示

---

## 7. 既知の制約・今後の拡張余地

- 通知はタブを開いている間のみ有効（プッシュ通知非対応）
- 定期タスクの「今日やるタスク」抽出は、タブを開いたタイミング・1分ごとの再描画で更新されるため、日付が変わった瞬間の即時反映は保証しない
- カテゴリは自由入力のため、表記ゆれ（例:「仕事」「Work」）の名寄せは行っていない
- 定期タスクの完了履歴は無期限に蓄積されるため、長期利用時のデータ量増加は将来的な検討事項
- 複数デバイス同期は第8章の方式に依存するため、端末・ブラウザによっては手動エクスポート／インポートのみ利用可能な場合がある

## 8. 複数デバイス間データ共有

### 8.1 採用しない方式（参考）

以下は本アプリの方針（1.3）と矛盾するため、**採用しない**。

| 方式 | 採用しない理由 |
|---|---|
| 自前バックエンド＋ユーザーアカウント | 公開後のデータ保管・削除請求・セキュリティ対応など運用負荷が発生する |
| Firebase / Supabase 等の BaaS | 開発者プロジェクト配下にユーザーデータが集約され、データ管理者の責任が生じる |
| Google Drive API / Dropbox API 等の OAuth 連携 | クラウドプロバイダへの OAuth アプリ登録・審査・スコープ管理が必要 |
| 開発者提供の同期サーバ | ユーザーデータのホスティングと可用性維持が必要 |

### 8.2 採用候補の比較

| 方式 | 概要 | 自動同期 | 主な対応環境 | 実装優先度 |
|---|---|---|---|---|
| A. 手動エクスポート／インポート | JSON ファイルの書き出し・読み込み | 手動 | 全ブラウザ | **Phase 1（必須）** |
| B. 同期ファイル（File System Access API） | ユーザーが指定した1ファイルを読み書き。OS／クラウドのフォルダ同期（iCloud Drive・Dropbox デスクトップ等）経由で他端末と共有 | 準自動※ | Chromium 系デスクトップ（Chrome / Edge） | **Phase 2（推奨）** |
| C. WebDAV | ユーザーが用意した WebDAV サーバ（Nextcloud・NAS 等）に JSON を PUT/GET | 準自動※ | `fetch` が使える環境（CORS 設定が必要） | **Phase 2（推奨）** |
| D. 同一 LAN 内 P2P 同期 | QR コードで WebRTC セッションを確立し、端末間で直接データ転送 | 手動起動 | WebRTC 対応ブラウザ | Phase 3（任意） |

※「準自動」＝アプリのタブを開いている間、一定間隔（例: 60秒）またはフォーカス復帰時にファイル／WebDAV を読み書きする。バックグラウンド常時同期は行わない。

**Phase 1 だけでも** AirDrop・メール添付・クラウドストレージの「ファイルとして共有」等で複数デバイス利用は可能。Phase 2 で日常利用の手間を減らす。

### 8.3 各方式の仕様

#### 8.3.1 A. 手動エクスポート／インポート

**エクスポート**
- ヘッダーまたは設定メニューから「データを書き出す」を実行
- 3.7 の同期ファイル形式で JSON を生成し、`<a download>` により `yaru-koto-dake-backup-YYYYMMDD.json` として保存
- Web Share API が利用可能な場合は共有シート経由での送信も可能（API 登録不要）

**インポート**
- 「データを読み込む」でファイル選択（`<input type="file">`）
- 読み込み前にプレビューし、マージ方式をユーザーに選択させる（8.5参照）
- インポート完了後、ローカルストレージを更新して画面を再描画

**利点**: 実装が単純、全環境で動作、審査・サーバ不要  
**欠点**: 端末間の反映はユーザー操作が必要

#### 8.3.2 B. 同期ファイル（File System Access API）

**初回設定**
1. ユーザーが「同期ファイルを選ぶ」を実行
2. `showOpenFilePicker`（既存ファイル）または `showSaveFilePicker`（新規作成）で JSON ファイルを1つ指定
3. 指定ファイルの親フォルダが iCloud Drive・Dropbox・Google Drive デスクトップアプリ等の同期対象であれば、**OS／クラウドクライアントがファイルを他端末へ複製**する（アプリは各クラウド API を呼ばない）

**同期動作**
- タブ表示中: 60秒間隔＋`visibilitychange`（タブ復帰）時に以下を実行
  1. ファイルを読み込み、リモートの `exportedAt` とローカルを比較
  2. 8.5 のマージルールで統合
  3. 統合結果をローカルストレージとファイルの両方に書き戻し
- ファイルハンドルの権限はセッション中のみ。再訪問時は `requestPermission` で再承認を求める

**制約**
- Safari / iOS では File System Access API の対応が限定的なため、**非対応時は A にフォールバック**し UI で案内する
- 同期の競合はマージルールに委ねる（8.5）

#### 8.3.3 C. WebDAV

**接続設定（ユーザー入力）**

| 項目 | 説明 |
|---|---|
| サーバ URL | 例: `https://cloud.example.com/remote.php/dav/files/username/` |
| パス | 同期ファイルのパス（例: `yaru-koto-dake/sync.json`） |
| ユーザー名 / パスワード | Basic 認証。アプリパスワードの利用を推奨（ヘルプテキストで案内） |

認証情報は **sessionStorage** にのみ保持し、永続化しない（タブを閉じれば消える）。毎回入力が煩わしいユーザー向けに「この端末に記憶する」オプションを **localStorage に暗号化して保存**するかは Phase 2 実装時に検討（パスフレーズによる端末ローカル暗号化を推奨）。

**同期動作**
- B と同様の間隔・トリガーで PROPFIND/GET/PUT を実行
- リクエスト先はユーザー指定のサーバのみ。開発者のインフラは経由しない

**制約**
- WebDAV サーバ側で CORS および `Authorization` ヘッダを許可する設定が必要（Nextcloud 等は管理者／ユーザーが設定可能）
- サーバ障害時はローカルデータのみで継続利用可能とし、同期失敗をトーストで通知

#### 8.3.4 D. 同一 LAN 内 P2P 同期（任意・将来）

- 「近くの端末と同期」から QR コードを表示
- もう一方の端末が QR を読み取り、WebRTC DataChannel で SDP を交換（**シグナリング用の自前サーバは不要**）
- 接続確立後、3.7 形式の JSON を一括送信し、受信側でマージ
- 常時同期ではなく、ユーザーが明示的に実行するワンショット同期

### 8.4 UI 要件（同期関連）

ヘッダーまたは設定パネルに以下を配置する。

| UI 要素 | 説明 |
|---|---|
| 同期ステータス | 「同期オフ」「同期済み（日時）」「同期待ち」「競合あり」等を表示 |
| 今すぐ同期 | B / C 有効時に手動で読み書きを実行 |
| データを書き出す / 読み込む | A の手動操作 |
| 同期の設定 | 同期方式の選択・接続設定・解除 |
| 同期の説明 | 「データはあなたの端末と、あなたが選んだファイル／サーバにのみ保存されます」旨の短文 |

### 8.5 マージ・競合解決

複数端末で同時に編集された場合の統合ルール。

**タスク（Task）**
1. `id` が一致するレコード同士を比較
2. 一方にのみ `deletedAt` がある場合: より新しい `deletedAt` を採用（削除の伝播）
3. 両方存在する場合: `updatedAt` が新しい方を採用（Last-Write-Wins）
4. 片方にのみ存在する場合: そのまま取り込む

**完了履歴（Completions）**
- 同一キーについて、記録日時（`at`）が新しい方を採用
- 値が「未完了（キー削除）」を表す場合は、明示的な削除マーカーを将来検討。Phase 1 では完了記録の追加のみマージ

**通知設定（Prefs）**
- `updatedAt` を Prefs に追加し、新しい方を採用

**インポート時のユーザー選択**
- 「リモートで上書き」「ローカルで上書き」「マージ（推奨）」の3択を提示
- 初回インポート（ローカルが空）の場合は確認なしで取り込み

**競合の検出**
- 同一 `id` のタスクで、両端末の `updatedAt` の差が同期間隔以内かつ内容が異なる場合を「競合候補」とし、マージ後にトーストで「○件の競合を新しい更新で解決しました」と通知（詳細一覧は Phase 3）

### 8.6 セキュリティ・プライバシー

| 項目 | 方針 |
|---|---|
| データの所在 | ユーザーのブラウザ・ユーザー指定ファイル・ユーザー指定 WebDAV のみ |
| 通信 | WebDAV は HTTPS のみ許可 |
| 暗号化 | Phase 1 では平文 JSON。Phase 2 以降でオプションのパスフレーズ暗号化（AES-GCM）を検討し、ファイル／WebDAV に保存する前にクライアント側で暗号化 |
| 開発者のデータアクセス | なし（テレメトリ・クラウドバックアップも送らない） |

### 8.7 実装ロードマップ

| フェーズ | 内容 | 成果 |
|---|---|---|
| Phase 1 | 手動エクスポート／インポート、`updatedAt` / `deletedAt` 追加、マージロジック | 全環境で端末間コピーが可能 |
| Phase 2 | 同期ファイル（File System Access API）または WebDAV のいずれかを先行実装 | 日常利用での手動コピーが不要に |
| Phase 3 | 残りの同期方式、P2P、暗号化オプション、競合詳細 UI | 利便性・セキュリティの強化 |

### 8.8 環境別の想定利用パターン

| 利用シーン | 推奨方式 |
|---|---|
| PC（Chrome）＋ スマホ（Safari） | PC で Phase 2 の同期ファイルを iCloud Drive 等に置き、スマホでは Phase 1 のインポートで随時反映（またはスマホ側は閲覧専用端末と割り切る） |

| 同一場所で一度だけ移す | P2P（Phase 3） |
